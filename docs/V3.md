# Byteling v3 — the modular pet platform

**Shipped.** v3 turned Byteling from a single hardcoded creature into a platform:
pets are swappable modules, they move with real physics, they wander with
manners, and the six bugs that dogged the v2 release are gone.

This document records what was built, why it was built that way, and the lessons
worth keeping. It supersedes the planning doc at [V3_PLAN.md](V3_PLAN.md).

---

## Part 1 — The pet platform

### A pet is a bundle of modules

The core refactor. A pet stopped being one component and became data plus
swappable parts:

```
Pet = {
  identity:    name, bio, unlock state, price (price unused for now)
  renderer:    HOW it's drawn — procedural SVG today, sprite-sheet later
  animations:  named states (idle, moving, sleeping)
  personality: voice profile shaping the LLM prompt + canned lines
  movement:    roaming behaviour
}
```

Plus a cross-cutting **effects layer** (the trail) that reads position and
velocity and works for any pet.

**The layout:**

| File | Role |
| --- | --- |
| `src/pets/types.ts` | The contracts: `Character`, `PetRenderState`, `Action`, `Facing`, `RendererKind` |
| `src/pets/registry.ts` | The character list — data, not code |
| `src/pets/PetView.tsx` | The one seam the app renders through; dispatches by renderer kind |
| `src/pets/renderers/` | One component per renderer kind |
| `src/effects/TrailEffect.tsx` | Renderer-independent particle trail |
| `src/roaming.ts` | The wandering brain |

**Why it matters:** the app calls `PetView` with a character and a render state
and never branches on which pet is active. Adding a companion is a registry
entry plus a renderer. Adding sprite-based pets later needs no caller changes —
that's the whole point of the interface.

The active character lives in Rust (`active_character.txt` in the app config
dir) with `get_active_character` / `set_active_character` commands and an
`active-character-changed` event, so the clinic's shop and the overlay stay in
sync across separate webviews and the choice survives restarts.

### Motion and inertia

The overlay samples window position every frame during a drag; the delta is
velocity. That velocity flows `App → PetView → CreatureRenderer → Creature`,
which deforms: the body leans, skews and stretches into travel while the limbs
slant backward with inertia.

The same velocity pipe carries roaming, so autonomous movement got inertia and
trails for free. That's the payoff of the module split — motion is a property of
velocity, not of *why* the pet is moving.

### The trail

`TrailEffect` watches velocity and spawns glowing motes in the pet's hue that
drift backward and fade. It knows nothing about which pet is active, so it works
for all of them and will work for every future one. On/off is persisted and
toggleable in settings.

## Part 2 — Roaming, and the restraint principle

**The design brief we set ourselves: a well-mannered cat, not a screensaver.**

An autonomous pet that paces across your screen while you work is a pet people
uninstall. So the rails were built *before* the autonomy:

- **It only wanders when you're actually away.** Idle detection lives in Rust
  (`cursor_idle_ms`), tracking real cursor movement anywhere on screen.
- **It hugs the screen edges.** Targets are always on the perimeter, never the
  middle where your windows and your attention are.
- **It rests far more than it moves.** Rest is the default phase; wandering is
  the exception.
- **The user always wins.** Grabbing the pet interrupts the plan instantly.
- **Never in fullscreen.** Games and videos are left alone.
- **Off by default.** Users opt in: Still, Calm, or Playful.

`RoamBrain` (`src/roaming.ts`) is a self-contained state machine — phases,
target selection, easing, and all timing decisions. `App.tsx` only drives it and
enforces the rails. Keeping the decisions in one testable place is what made the
behaviour tunable instead of mysterious.

**A lesson worth keeping:** the first implementation ran the wander/rest decision
every frame. At 60fps it re-rolled into a fresh rest window constantly and the
pet never moved at all. Decisions belong in a state machine that advances once
per cycle, not in the render loop.

## Part 3 — The six v2 bugs, and what they taught

### 1. The pet stole clicks (bugs 5 + 6)

**Symptom:** an invisible region that swallowed clicks, and clicks in the pet's
transparent corners hitting the pet instead of the app behind. Bad everywhere,
unacceptable while gaming.

**Root cause:** one hardcoded rectangle in the Rust cursor watcher decided
interactivity.

**Why the obvious fixes failed:** we tried a smaller rectangle, then a circle
around the body. Both are wrong in principle — every pet has a different shape,
and the menu and speech bubble deliberately live *outside* the pet's body. No
fixed region in Rust can be right.

**The fix:** invert the responsibility. The frontend knows the shape, so the
frontend decides. It computes its live interactive regions — the pet's real ink
bounds via `getBBox()`, plus any open menu, confirm panel or bubble — and
publishes them to Rust (`set_hit_rects`). Rust stores them in a `Mutex<Vec<...>>`
and tests the cursor against them in window-relative logical pixels. Correct for
any pet, any UI, any display scale.

**Lesson:** when a fix needs knowledge one layer doesn't have, move the decision
rather than improving the guess.

### 2. The pet went silent

**Root cause:** the prompt told a reasoning model to reply "in ONE short line
under 18 words, no quotes, at most one emoji." It dutifully spent all 512 tokens
counting words in its reasoning trace and returned empty content with
`finish_reason: length`.

**The fix:** remove the countable rules. "Answer immediately with one short line
— no deliberation. Keep it brief." Nothing to audit, nothing to deliberate over.

**Lesson:** constraints a model can *check* are constraints it will check, at
your expense. Reasoning models need prompts that don't invite an audit.

### 3. Reactions were slow

**Root cause:** foreground detection polled every 2 seconds, so noticing an app
change took up to 2s before the LLM had even started.

**The fix:** a Win32 event hook — `SetWinEventHook` with
`EVENT_SYSTEM_FOREGROUND`, its own message pump on a dedicated thread. Windows
calls us the instant focus changes. Reactions are immediate and the app burns no
CPU idling.

The callback must be `extern "system"` and can't capture state, so the notifier
lives in a `OnceLock`. Deduping also moved from category to executable, so
switching between two browsers now registers and the pet can name the app.

### 4. It didn't know your apps

Expanded from a dozen executables to ninety, with substring matching so
versioned names resolve. More importantly, unknown apps now get a readable
fallback (`my_cool_app.exe` → "My Cool App") rather than a mangled string — the
pet should never be *confidently wrong* about what you're doing.

### 5. It didn't appear on launch

**Symptom:** in packaged builds only, the pet was often invisible until you
closed and reopened a few times.

**Root cause:** `if (!species) return null` gated the entire overlay on one async
call. In packaged builds the webview can be ready before Rust's `setup()` has
managed the app state, so the call rejected, `species` stayed null, and the pet
never rendered. Relaunching sometimes won the race — hence the symptom.

**The fix:** retry the fetch, guard every other startup call so one rejection
can't skip the rest, and never render `null`. An invisible overlay is
indistinguishable from a crash.

**Lesson:** dev servers hide race conditions. Timing bugs must be reproduced in
the packaged build.

## Part 4 — Engineering notes

Things that cost real time and are worth remembering:

- **A Rust "function is never used" warning on a `#[tauri::command]` means it
  isn't in `invoke_handler`.** The frontend's `invoke` then throws and the
  feature is silently dead. The compiler tells you; read it.
- **`#[tauri::command]` functions, statics and `use` must be top-level** — nested
  inside the setup closure they can't be seen by `generate_handler!`.
- **Empty `catch {}` blocks hide the bug you're hunting.** Several of these
  swallowed the real errors for hours. Log, always.
- **CSS animations override inline styles.** The `breathe` keyframe silently
  clobbered the motion transform, and separately clobbered the creature's
  life-stage scale via `scale(var(--s, 1))` with `--s` undefined — Elder and
  Sturdy sizing had never once applied. Fixed by separating the motion group and
  passing scale as a CSS variable.
- **Explicit geometry beats rotation.** Making limbs trail via `rotate()` failed
  through several sign flips. Drawing them as slanted paths where the foot is
  computed as `topX - limbTrail` made the direction unambiguous and it worked
  first try.
- **`tauri build` runs strict `tsc`** and catches dead code that dev mode
  ignores. Expect a cleanup pass before every release build.
- **Version bumps need three files:** `tauri.conf.json`, `Cargo.toml`
  (`[package]`), and `package.json`. Miss one and the installer is misnamed.

## Part 5 — What's next

Deferred to v4:

- **Smarter AI.** The pipeline is solid; the responses could be better.
- **The dizzy threshold.** Instant per-app detection made a single app switch
  trip the dizzy state. It should need several switches in a short window.
- **Branding.** A proper wordmark lockup.
- **Persistence and hatch date** — unlocks age, care history, and the weekly
  checkup letter.
- **All-drives storage panel** — vertical bars for each drive, like a stomach.
- **Sprite-based pets** — the renderer interface already supports them; they need
  art.
- **Autostart** — cut from v2, still unfinished.
- **A signed installer**, so SmartScreen stops scaring people.
